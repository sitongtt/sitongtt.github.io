Spring Bean的生命周期

加载类-实例化-初始化-销毁(可以拓展，但大概就这四个)

Spring Bean实例化三种方式

构造器、静态工厂、非静态工厂
工厂模式：虽然降低了一定的耦合度，但要进行大量的new工作
IoC容器-BeanFactory:

  1. 尽可能晚的初始化Bean
  2. 当applicationContext.xnl文件中没有调用里面的bean时，他就不实例化(懒加载)
  3. 解析xml时不校验（可能出现问题）
    IoC容器-ApplicationContenxt:
  4. 尽可能早的初始化Bean
  5. String…代表后面可以有任意多个字符串
  6. 执行调用钱，容器的中的所有的bean都实例化了
  7. 不论调用与否，applicationContenxt都会实例化工厂里的bean
  8. 解析xml时顺便校验
    两者的区别
    在服务启动时ApplicationContenxt就会校验，所以不会产生运行时bean装配错误；相反BeanFactory获取bean时，可能会产生异常。

Spring Bean的定义和命名

class：实例化对象的类型，通常必填。

name：通过id或name属性来指定bean,通常用id来保证唯一性。

scope:定义作用域。

constructor arguments:构造器注入。

autowiring mode：可以指定自动装配（autowire）相互协作bean之间的关联关系（5种类型）

dependency checking mode：对容器中bean的依赖设置进行检查 ，可以检查bean定义中实际属性值的设置。

lazy-initialization mode（懒加载）：如果你不想让一个singleton bean在ApplicationContext实现在初始化时被提前实例化，那么可以将bean设置为延迟实例化。一个延迟初始化bean将告诉IoC 容器是在启动时还是在第一次被用到时实例化。在XML配置文件中，延迟初始化将通过元素中的lazy-init属性来进行控制”true/false”.

Spring Bean作用域

  1. singleton：单例模式，
    Spring IoC容器中仅存在一个Bean实例。Singleton作用域是Spring中的缺省作用域
  2. prototype:原型模式，
    每次从容器获取bean时，容器都将创建一个新的Bean实例。
  3. request:
    每一次Http请求都会创建一个新的Bean，仅在当前Http Request内有效。
  4. session:
    同一个Http Session共享一个Bean,不同的Session请求则会创建新的实例，该bean实例仅在当前Session内有效。
  5. global Session：
    在一个全局的Http Session中，容器会返回该Bean的同一个实例，仅适用于Portlet应用环境。

依赖注入

构造函数注入、属性注入、接口注入（侵入性强，Spring不提供）

构造函数注入
<bean id="" class="">
    <constructor-arg name="" ref="填构造方法括号里的" value=""></constructor-arg>
</bean>
别忘了在类里写构造函数
属性注入

通过set方法参数注入

<bean id="" class="">
    <property name="" ref="填set后面的参数"></property>
</bean>
属性标签

value：基本类型（包装类型）或String类型
ref：引用类型，容器中其它的bean
list：List或数组类型 

<list>
    <value></value>
或者
    <ref bean="" />
</list>

set：Set类型。用法与list类似。 

<set>
    <value></value>
或者
    <ref bean="" />
</set>

map：Map类型 

<map>
    <entry key="">
        <ref bean="" />
    </entry>
</map>

props：Properties类型，键值为String类型的，所以直接写值。 

<props>
    <prop key="">something</prop>
    <prop key="">something</prop>
</props>
